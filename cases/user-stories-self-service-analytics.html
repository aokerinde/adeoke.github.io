01 · Background

Why this backlog exists

Data lived across disconnected marts, warehouses, and cloud platforms, so business and operational users couldn't trust reporting or self-serve answers without routing every request through a central data/BI team. That team became a bottleneck not because the requests were hard, but because there was no governed, trusted layer non-technical users could query directly. This backlog structures the work to fix that: consolidate the sources, build a governed semantic layer on top, then hand business users a self-service experience they can trust without a ticket queue in between.

03 · Backlog

Epic → Feature → Story, with acceptance criteria

Three epics structured the rollout: build the unified integration layer, enable the self-service experience on top of it, and drive adoption while proving the central team's workload actually dropped. Each epic breaks into features, and each feature into backlog-ready stories.

Epic 1Unified Data Integration Layer
Consolidate marts, warehouses, and cloud sources into a single governed pipeline with standardized definitions.
Feature 1.1Cross-Source Pipeline Consolidation
Replace point-to-point integrations with a single governed ingestion pipeline.
1.1.1Must
Primary user: Data Engineer

As a Data Engineer, I want all source marts, warehouses, and cloud platforms consolidated into a single governed ingestion pipeline, so that I maintain one pipeline instead of a growing set of one-off integrations per report request.

Acceptance Criteria
  • Given a new source system is onboarded, when it is connected, then it flows through the same governed ingestion pipeline as existing sources rather than a bespoke integration.
  • Given the pipeline runs on schedule, when it completes, then ingestion success/failure and row-level reconciliation counts are logged for every source.
1.1.2Must
Primary user: Data Engineer

As a Data Engineer, I want automated data quality validation on every ingestion run, so that bad data is caught before it reaches a business user's report.

Acceptance Criteria
  • Given data is ingested, when validation rules run, then schema conformance, null-rate thresholds, and referential integrity are checked automatically.
  • Given a validation rule fails, when the failure is detected, then downstream datasets built from that source are flagged as stale rather than silently updated with bad data.
Feature 1.2Semantic Layer & Governance
Standardize metric definitions so the same term means the same thing across every report.
1.2.1Must
Primary user: BI/Data Governance Lead

As a BI/Data Governance Lead, I want a certified semantic layer with standardized metric definitions, so that "cycle time" or "revenue" means the same thing regardless of which report a user builds.

Acceptance Criteria
  • Given a metric is defined in the semantic layer, when it is published, then it carries a single owner, definition, and calculation logic visible to all consumers.
  • Given two reports reference the same certified metric, when compared, then they return identical values for the same filter context.
1.2.2Should
Primary user: Business Analyst

As a Business Analyst, I want to see data lineage for any certified dataset, so that I can confirm where a number comes from before I present it.

Acceptance Criteria
  • Given a certified dataset, when a user views its details, then source system(s), last refresh time, and transformation summary are displayed.
Epic 2Self-Service Reporting Enablement
Give business users a governed workspace to build their own reports on certified data.
Feature 2.1Governed Self-Service Workspace
Role-based access to certified datasets, without exposing raw source systems.
2.1.1Must
Primary user: Business Analyst

As a Business Analyst, I want role-based access to a workspace of certified datasets relevant to my department, so that I can build reports without waiting on a data request ticket.

Acceptance Criteria
  • Given a user is assigned a department role, when they log into the workspace, then they see only the certified datasets relevant to that role.
  • Given a user attempts to access a dataset outside their role, when the request is made, then access is denied with a clear explanation and request path.
2.1.2Should
Primary user: Business Analyst

As a Business Analyst, I want a searchable library of certified datasets with descriptions, so that I can find the right dataset without asking the BI team what exists.

Acceptance Criteria
  • Given the dataset library, when a user searches by keyword, then matching datasets return with owner, description, and last-refresh metadata.
Feature 2.2Ad Hoc Report Builder Experience
Let users build and share reports without writing SQL.
2.2.1Must
Primary user: Business Analyst

As a Business Analyst, I want a drag-and-drop report builder on top of certified datasets, so that I can create a report without writing SQL or involving engineering.

Acceptance Criteria
  • Given a certified dataset, when a user drags fields into the builder, then a report renders using only pre-validated, governed fields.
  • Given a report is built, when the user saves it, then it can be shared with defined role-based permissions.
2.2.2Could
Primary user: Operations Manager

As an Operations Manager, I want guided templates for common report types (trend, comparison, breakdown), so that I don't start every report from a blank canvas.

Acceptance Criteria
  • Given the report builder, when a user selects a template, then it pre-configures a starting layout mapped to the user's available certified fields.
Epic 3Adoption, Trust & Reduced Central-Team Dependency
Drive adoption and prove the central reporting team's ad hoc request volume actually dropped.
Feature 3.1Training & Enablement
Get users comfortable self-serving instead of defaulting to a request ticket.
3.1.1Should
Primary user: Business Analyst

As a Business Analyst, I want hands-on onboarding and a champions network, so that I feel confident building my own reports instead of defaulting to a data request ticket.

Acceptance Criteria
  • Given a new workspace user, when they are onboarded, then they complete a guided walkthrough and are assigned a department champion contact.
3.1.2Could
Primary user: Business Analyst

As a Business Analyst, I want in-product help and documentation linked to each certified dataset, so that I can resolve questions without filing a support ticket.

Acceptance Criteria
  • Given a certified dataset, when a user clicks help, then contextual documentation and example reports are shown.
Feature 3.2Usage Monitoring & Trust Signals
Prove adoption is real and that users can trust what they're looking at.
3.2.1Must
Primary user: Central Reporting Team

As a member of the Central Reporting Team, I want a usage dashboard showing self-service adoption and ad hoc request volume over time, so that we can demonstrate the initiative actually reduced our request queue.

Acceptance Criteria
  • Given self-service usage occurs, when the dashboard refreshes, then active users, reports built, and dataset query volume are tracked weekly.
  • Given ad hoc request tickets are logged, when compared over time, then the dashboard shows request volume trending against self-service adoption.
3.2.2Should
Primary user: Business Analyst

As a Business Analyst, I want a visible data-freshness indicator on every report, so that I know whether I'm looking at current data before I share it.

Acceptance Criteria
  • Given a report is opened, when it renders, then the underlying dataset's last-refresh timestamp is displayed alongside the report.

These 12 stories are a representative sample of the full backlog used to run this initiative; the remainder followed the same Epic-Feature-Story structure across additional domains and edge cases.

04 · Definition of Ready / Done

How a story moved through the backlog

Definition of Ready

  • Story has a clear user, need, and benefit statement
  • Acceptance criteria are written and reviewed by engineering
  • Dependencies on other stories/features are identified
  • MoSCoW priority assigned relative to the current release

Definition of Done

  • All acceptance criteria pass in QA and UAT
  • Data quality/validation checks pass for any new dataset touched
  • Documentation or in-product help updated where applicable
  • Usage tracking is in place before the story is considered adopted, not just shipped
